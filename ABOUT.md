
Charter — это production-ориентированный FastAPI микросервис внутри системы GTS (travel). Его задача — быть быстрым слоем хранения и выборки чартерных предложений (flight offers), чтобы повторные поиски не нагружали основной GTS. Сервис не реализует поисковый движок, а сохраняет ответы GTS в PostgreSQL и возвращает их в исходном виде по запросу. Это снижает нагрузку на GTS, ускоряет ответы и позволяет инвалидировать предложения, если апстрим сообщает об их недействительности или бронировании.

Архитектура построена по принципу чистых слоёв (аналог NestJS, но на FastAPI):
- **router** — HTTP endpoints, только маршрутизация и Depends
- **service** — бизнес-логика, обработка данных, решения по обновлению/инвалидации
- **repository** — работа с БД через SQLAlchemy, не знает про HTTP и валидацию
- **schemas** — Pydantic-схемы для валидации и сериализации, преобразуют JSON в типизированные объекты
- **models** — ORM-модели SQLAlchemy
- **core** — конфиг, настройки, общие утилиты
- **db** — сессия SQLAlchemy, база

Используются: Python 3.11, FastAPI, Uvicorn, SQLAlchemy 2.x, asyncpg, Alembic (migrations), pydantic-settings, uv (env/deps), psycopg2-binary (для Alembic), PostgreSQL. Все зависимости управляются через pyproject.toml и uv.lock. Подключение к БД через DATABASE_URL из env.

**Модель данных:**
Каждое предложение (offer) — отдельная строка в таблице. Входящий payload — массив офферов, каждый сохраняется отдельно. В таблице: id (PK), provider_id, supplier_offer_id, origin, destination, departure_date, price, currency, is_active, expires_at, raw_json. raw_json хранит оригинальный payload для возврата в неизменном виде. Поиск, фильтрация, сортировка, дедупликация и инвалидирование идут по отдельным колонкам, а raw_json нужен для возврата клиенту.

**Поиск:**
Сервис не хранит сущность "search request" и не использует Redis для replay. Это именно storage-слой для офферов. Поиск — по origin, destination, departure_date (в будущем — пассажиры, класс, direct/connecting, авиалинии и т.д.). Начинаем с простого, расширяем без ломки дизайна.

**Soft delete:**
Инвалидированные офферы не удаляются физически сразу, а помечаются is_active = false. Физическое удаление — отдельный фоновый процесс. Все выборки фильтруют is_active = true и expires_at > now(). Это позволяет безопасно инвалидировать и избегать дубликатов.

**Upsert:**
Сохранение офферов — через ON CONFLICT upsert по (provider_id, supplier_offer_id), чтобы повторные записи были идемпотентны.

**Аутентификация:**
Сервис поддерживает JWT-авторизацию для внутренних и внешних клиентов. Для защищённых эндпоинтов используется JWT access token, который выдаётся после логина. Валидация токена реализована через Depends и отдельный dependency-слой, чтобы бизнес-логика не знала о деталях авторизации. Ключи и параметры токена хранятся в env через pydantic-settings.

**Валидация:**
Pydantic-схемы валидируют входные данные (например, EmailStr, сложность пароля через field_validator). Ошибки валидации и бизнес-ошибки унифицированы через кастомные exception handler'ы в main.py.

**Миграции:**
Alembic использует psycopg2-binary для sync-подключения, приложение — asyncpg. Проблема с несовпадением драйверов решена.

**Стиль:**
Проект строится с прицелом на production: чистые слои, строгая типизация, явное разделение ответственности, минимум магии, объяснения кода и решений. Все новые фичи внедряются поэтапно, без спешки и с акцентом на поддерживаемость.

**Основной бизнес-флоу:**
GTS возвращает массив офферов, сервис сохраняет каждый как отдельную строку, затем по запросу возвращает подходящие офферы из PostgreSQL, сохраняя оригинальный JSON. Дубликаты исключаются через уникальный индекс, инвалидирование — через is_active. Сервис — это быстрый кэш и storage для экосистемы GTS.
